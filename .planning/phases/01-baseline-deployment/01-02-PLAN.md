**Automated checks the executor should run first:**

1. Verify the site is reachable over HTTPS:
   ```bash
   curl -sI https://vbi.narenlife.com | head -10
   ```
   Expected: HTTP/2 200 response. If connection refused or cert error, certificate is still provisioning -- wait and retry.

2. Check HTTP-to-HTTPS redirect:
   ```bash
   curl -sI http://vbi.narenlife.com | head -5
   ```
   Expected: HTTP/1.1 301 with Location header containing "https://vbi.narenlife.com". If this returns 200 (no redirect), "Enforce HTTPS" is not yet enabled.

3. Check for mixed content in the HTML (the only http:// references should be SVG namespaces):
   ```bash
   curl -s https://vbi.narenlife.com | grep -o 'http://[^"'"'"' <]*' | grep -v 'www.w3.org' | head -10
   ```
   Expected: no output. The only http:// URL in the site is `http://www.w3.org/2000/svg` which is a namespace URI, not a resource load.

4. Spot-check a QA fix is live (rel=noopener present):
   ```bash
   curl -s https://vbi.narenlife.com | grep -c 'rel="noopener"'
   ```
   Expected: a number greater than 0.

**If HTTPS is not yet working (curl fails or returns cert error):**

The user must visit https://github.com/narulraj283/vbi-website/settings/pages and check:
- Custom domain shows "vbi.narenlife.com" with a green "DNS check successful" indicator
- "Enforce HTTPS" checkbox is available and checked

If "Enforce HTTPS" is grayed out:
1. Wait up to 1 hour for certificate provisioning
2. If still grayed out after 1 hour: remove the custom domain from the settings field, wait 30 seconds, re-add "vbi.narenlife.com", and save. This restarts certificate provisioning.
3. Check GoDaddy for conflicting A records on the "vbi" subdomain (there should be none -- only the CNAME)
